# 用户优先指令

## 目的

本文件用于记录项目所有者对 `vscode-acp-iot` 的当前明确指示。

当存在以下情况时，应优先参考本文件：

- 开放性方案有多种可选路径
- 文档中的历史路线与当前意图不完全一致
- 需要在多个可行方案中做取舍
- 需要判断“当前阶段更应该做什么”

---

## 优先级说明

当出现冲突时，按以下优先级执行：

1. 用户在当前对话中的最新明确要求
2. 本文件中的明确指示
3. `.claude_dev/plan/` 中的当前执行计划
4. `.claude_dev/roadmap/` 中的路线图与版本规划
5. 其他历史文档、研究材料和旧方案

如果本文件与用户最新消息冲突，应以**用户最新消息**为准。

---

## 当前使用规则

- 对于开放性问题，不要默认选择“最通用”的方案，应优先选择**更符合本文件意图**的方案。
- 对于路线图、版本规划、结构设计、优先级判断，应把本文件视为当前阶段的主决策输入之一。
- 如果本文件没有覆盖某个具体问题，再回退到 `plan/`、`roadmap/` 和其他资料。
- 如果发现本文件内容已经过时，应提醒用户更新，而不是自行长期沿用旧判断。
- 在自己开发时，自己完善 `plan/`、`doc/`、`codemap/`、`version/`。

## 当前明确指示

### 1. 核心任务目标
- 在 `vscode-acp` 插件中，新增并集成一个名为 "EMBEDDER MONITOR" 的**侧边栏视图 (Webview View)**。
- 视觉风格与 UI 排版必须高度复刻设计图：包含多端口标签、连接/过滤/清空/波特率探测控制按钮、日志文本框、命令行输入框、端口及波特率和换行符选项栏。

### 2. 双重集成级别
- **级别一 (用户手动控制)**：用户能通过侧边栏 UI 选择真实串口、连接并直接发送命令及查阅日志。
- **级别二 (AI Agent 控制)**：必须实现 AI Agent 对串口的完全控制能力。扩展标准 ACP (Agent Client Protocol) 协议，支持 `acp:serial_connect`、`acp:serial_disconnect`、`acp:serial_write` 接口，并建立串口 `onData` 向 Agent 实时反馈的通道 (`acp:serial_data`)。

### 3. 系统弹性与稳定性
- **二进制依赖与 Webpack 拦截**：针对物理串口库 `serialport`，绝不能让 Webpack 将其打包（会导致 C++ 扩展崩溃），必须在 `webpack.config.js` 中将其作为 `externals` 提出。
- **降级容灾与零崩溃**：底层必须设计懒加载 (`try-catch require`)，当二进制包加载失败时，自动静默降级到虚拟硬件模拟器 (`VirtualSerialSimulator`)，保障全平台环境随时可以顺利激活和测试 Agent 通信闭环。
- **状态持久与资源防抢**：由于支持多个 Agent（如同时存在 Copilot 和 Claude），`SerialManager` 必须要实现为单例 (Singleton)，防止争夺同一个物理串口口；另外针对 VS Code 侧边栏折叠会销毁 Webview 的原生限制，必须在 TS 端通过环形缓冲区 (Ring Buffer) 暂存日志，并在 Webview `ready` 时全量回显。

### 4. 版本与规划一致性
- 每一个阶段/版本的具体开发计划（如 v0.3.0）必须严格做到在 `plan/<version>.md` 与 `version/<version>.md` 中具有一比一的对称管理。
- 项目迭代进展、路线图变化需随时在 `roadmap/ROADMAP.md` 及 `codemap/README.md` 中跟进反映。
